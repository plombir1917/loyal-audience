# vk-loyal-users-parser

Сервис на Go для сбора данных из открытых сообществ ВКонтакте (Чувашская
Республика): посты, комментарии, лайки и агрегат реакций. Классифицирует
тональность комментариев и постов, считает ядро лояльной аудитории и
материализует статистику для AdminJS.

## Структура

```
cmd/app                  — точка входа, сборка слоёв и запуск пайплайна
internal/core/config     — конфигурация из переменных окружения
internal/core/logger     — структурированный логгер (slog)
internal/model           — доменные сущности (общие для слоёв)
internal/vk              — клиент API ВК (сообщества, посты, лайки, комментарии, реакции)
internal/classifier      — классификация тональности (LLM или словарный fallback)
internal/storage         — запись данных и материализация статистики в Postgres (pgx)
internal/service         — оркестрация сбора, классификации, ядра и статистики
```

## Хранилище

Данные пишутся в тот же Postgres, что использует `admin-panel`. **Схемой и
миграциями владеет Prisma** (`admin-panel/prisma/schema.prisma`) — перед первым
запуском парсера применить миграции:

```
cd ../admin-panel && npx prisma migrate deploy
```

Запись идемпотентна (upsert по первичным ключам), повторный прогон не плодит
дубли. Сбор инкрементальный: сообщества и посты сканируются каждый прогон (чтобы
подхватывать новые публикации), а уже сохранённые комментарии и лайки
пропускаются — классификатор не дёргается повторно. Прерванный прогон при
перезапуске дозабирает недостающее, не переобрабатывая готовое. Флаг
`REPARSE_EXISTING=true` отключает все пропуски и переобрабатывает уже
сохранённые сущности заново (в т.ч. переклассифицирует посты и комментарии),
перекрывая `SKIP_EXISTING_COMMUNITIES`. AdminJS отображает собранные данные
(разделы community / post / comment / like / user).

### Охват

Регион разрешается в полный список городов через `database.getCities`
постранично (у Чувашии ~1728 населённых пунктов) — по каждому идёт подзапрос
`groups.search`. Полный дамп городов лежит в `data/chuvashia_cities.json`. Для
максимального охвата по умолчанию нет лимитов на число сообществ
(`MAX_COMMUNITIES=0`) и на число постов в сообществе (`MAX_POSTS_PER_GROUP=0` —
все посты с `COLLECT_SINCE`). Это заметно дольше и упирается в лимит VK
(`VK_RATE_LIMIT`, ≈3 req/s).

## Классификатор

Основной классификатор — реальная модель через OpenAI-совместимый
chat/completions endpoint (OpenAI, OpenRouter, Ollama, llama.cpp и т.п.).
Настраивается через env: `LLM_BASE_URL`, `LLM_MODEL`, `LLM_API_KEY`. Замена
модели — только через env, без изменения логики (контракт — интерфейс
`classifier.Classifier`).

Встроенный словарный классификатор работает как **fallback**: он включается
автоматически, если вызов модели завершился ошибкой (нет ключа, недоступна сеть,
лимиты), а также используется единственным, если `LLM_BASE_URL` не задан. Так
пайплайн не прерывается из-за сбоя классификатора.

Классифицируются и комментарии, и тексты постов. Вызовы модели — самая медленная
часть, поэтому они идут конкурентно пулом горутин (`CLASSIFY_CONCURRENCY`, по
умолчанию 8); запись в БД остаётся последовательной. VK-вызовы не
распараллеливаются — их держит `rate.Limiter` (`VK_RATE_LIMIT`).

## Реакции

Агрегат реакций поста (эмодзи) забирается инлайн из `wall.get` (нужна версия API
≥ 5.199) и складывается в таблицу `reaction` по типам. Тональность реакции:
обычный лайк (id 0) — нейтральный и учитывается отдельно таблицей `likes`;
сердце/восторг/смех/удивление — позитивные; печаль/гнев — негативные. VK отдаёт
только суммарные счётчики по объекту, привязки реакции к пользователю нет.

## Ядро аудитории

`segment` пересчитывается по формуле ядра: `is_core = (лайков > LIKE_THRESHOLD)
[AND|OR] (позитивных комментариев > COMMENT_THRESHOLD)` (операция — `CORE_COMBINE`,
по умолчанию `and`). Сегмент: `ядро → Loyal`; иначе `есть негативный комментарий
→ Disloyal`; иначе `Neutral`. У пользователя сохраняются метрики вовлечённости
(`like_count`, `comment_positive/negative/neutral`, `is_core`).

## Статистика

После сбора материализуются таблицы `stats_*` (пересчёт каждый прогон, видны в
AdminJS): `stats_sentiment_by_reaction` — распределение тональностей комментариев
по доминирующей реакции поста; `stats_post_sentiment_map` — связь тональности
комментариев/лайков с тональностью поста; `stats_core` — доля ядра в двух
вариантах (`likes_only`, `likes_plus_comments`).

## Конфигурация

См. `.env.example`. Обязательны `VK_TOKEN` и `DATABASE_URL`.

## Запуск

```
make run          # локально (нужны экспортированные переменные окружения)
make build        # бинарь в bin/app
make test         # юнит-тесты (классификатор)
```

В Docker (пакетный прогон, из корня репозитория):

```
docker compose --profile parser run --rm parser
```
