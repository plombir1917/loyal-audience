# Техническое задание (актуальная редакция)
## Система мониторинга и сегментации аудитории ВКонтакте (Чувашская Республика)

> Обновлено по факту реализации. Отражает текущую версию проекта: сбор реакций
> с тональностью, тональность постов, формулу «ядра» лояльной аудитории и
> материализованную статистику в AdminJS.

---

## 1. Цель проекта

Сервис на Go для автоматического сбора данных из открытых сообществ ВКонтакте,
относящихся к Чувашской Республике, с последующей классификацией пользователей
по признаку **лояльности** к анализируемым информационным поводам и публикациям.

Итоговая метка пользователя учитывает не случайный единичный лайк, а совокупность
сигналов: **лайки**, **реакции-эмодзи**, **тональность комментариев** и
**тональность содержания постов**.

---

## 2. Архитектура

Монорепозиторий из двух сервисов поверх общей БД PostgreSQL:

| Компонент | Стек | Роль |
|-----------|------|------|
| `vk-parser` | Go 1.26 | Сбор данных из VK API, классификация тональности, расчёт ядра и статистики |
| `admin-panel` | NestJS + AdminJS + Prisma | Владелец схемы БД и миграций; веб-панель просмотра данных и статистики (read-only) |

**Схемой и миграциями владеет Prisma** (`admin-panel/prisma/schema.prisma`).
Go-сервис зеркалит модель и пишет данные идемпотентными upsert'ами; перед первым
запуском парсера применяются миграции (`npx prisma migrate deploy`).

Сбор **инкрементальный и идемпотентный**: сообщества и посты сканируются каждый
прогон (чтобы подхватывать новые публикации), уже сохранённые комментарии/лайки и
уже классифицированные посты пропускаются — классификатор не дёргается повторно.
Флаг `SKIP_EXISTING_COMMUNITIES` позволяет пропускать целиком уже собранные
сообщества на самом первом этапе.

## 3. Функциональные требования

### 3.1. Сбор сообществ
Получение списка открытых сообществ ВКонтакте и отбор связанных с Чувашской
Республикой. Фильтрация по региону сообщества и указанному населённому пункту
(разрешение региона → города → поиск сообществ по городу и ключевым словам).

### 3.2. Сбор публикаций
Для каждого сообщества — последние публикации за выбранный период (с начала года,
параметр `COLLECT_SINCE`), с ограничением `MAX_POSTS_PER_GROUP`.

### 3.3. Сбор пользователей
Для каждого поста собираются:
- **лайки** (`likes.getList`) — пользователи, поставившие лайк; лайк считается
  сигналом потенциально лояльной аудитории;
- **комментарии** (`wall.getComments`) — пользователи, оставившие комментарии;
- **агрегат реакций-эмодзи** по посту (см. 3.5).

### 3.4. Классификация тональности комментариев
Каждый комментарий классифицируется по категориям **positive / negative /
neutral**.

- Основной классификатор — реальная LLM через OpenAI-совместимый
  `chat/completions` endpoint (OpenAI / OpenRouter / Ollama / llama.cpp).
  Настраивается через env (`LLM_BASE_URL`, `LLM_MODEL`, `LLM_API_KEY`), **замена
  модели без изменения логики** (контракт — интерфейс `classifier.Classifier`).
- Встроенный **словарный классификатор** — fallback: включается автоматически при
  сбое/недоступности модели, либо как единственный, если LLM не настроена. Сбой
  классификатора не прерывает пайплайн.
- Классификация идёт **конкурентно** пулом горутин (`CLASSIFY_CONCURRENCY`,
  по умолчанию 8); запись в БД остаётся последовательной. VK-вызовы удерживаются
  `rate.Limiter` (`VK_RATE_LIMIT`, ≈3 req/s).

### 3.5. Сбор реакций-эмодзи с тональностью *(задача 1)*
Для каждого поста собирается **агрегат реакций** (эмодзи) — приходит инлайн из
`wall.get` (требуется версия VK API ≥ 5.199) и складывается в таблицу `reaction`
по типам реакции.

Тональность реакции задаётся статической (конфигурируемой) таблицей
`vk_reaction_id → sentiment`:

| Реакция | vk_reaction_id | Тональность |
|---------|:--------------:|-------------|
| Обычный лайк | 0 | neutral (учитывается отдельно таблицей `like`) |
| Сердце | 1 | positive |
| Огонь / восторг | 2 | positive |
| Смех (haha) | 3 | positive |
| Удивление (wow) | 4 | positive |
| Печаль (sad) | 5 | negative |
| Гнев (angry) | 6 | negative |

> Ограничение VK: реакции отдаются **только суммарным счётчиком по объекту**;
> привязки конкретной реакции к пользователю API не предоставляет.

### 3.6. Классификация тональности постов *(задача 3)*
Текст каждого поста классифицируется тем же классификатором (positive / negative
/ neutral) и сохраняется в `post.sentiment`. Уже классифицированные посты
пропускаются при повторных прогонах.

## 4. Ядро лояльной аудитории *(задача 4)*

Метка пользователя определяется формулой «ядра», заменяющей прежнюю простую
сегментацию:

```
is_core = (like_count > LIKE_THRESHOLD)  [AND | OR]  (comment_positive > COMMENT_THRESHOLD)
```

- Операция объединения — параметр `CORE_COMBINE` (`and` | `or`). По умолчанию
  `and` (строгое ядро). ТЗ допускает дизъюнкцию — переключается без изменения
  кода, тюнится по статистике.
- Предварительные пороги: `LIKE_THRESHOLD = 1`, `COMMENT_THRESHOLD = 1`.

**Сегмент** пользователя выводится из ядра:

| Условие | Сегмент |
|---------|---------|
| `is_core = true` | **Loyal** — поддерживающая аудитория |
| иначе, если есть хотя бы один негативный комментарий | **Disloyal** — критически настроенная |
| иначе | **Neutral** — неопределившаяся |

По каждому пользователю сохраняются **метрики степени и характера
вовлечённости**: `like_count`, `comment_positive`, `comment_negative`,
`comment_neutral`, `is_core`, `segment`.

Расчёт (`RecomputeUsers`) выполняется одним SQL-проходом в конце прогона.

## 5. Статистика *(задачи 2, 4Б, 5)*

После сбора материализуются таблицы `stats_*` (пересчёт каждый прогон, видны в
AdminJS в разделе «Статистика»):

| Таблица | Задача | Содержание |
|---------|:------:|------------|
| `stats_sentiment_by_reaction` | 2 | Распределение тональностей комментариев по доминирующей реакции поста (бакеты positive / negative / neutral / none) |
| `stats_post_sentiment_map` | 4А | Связь тональности комментариев и лайков с тональностью поста |
| `stats_core` | 4Б, 5 | Доля ядра в двух вариантах расчёта — итоговая сводка |

**Две таблицы расчёта ядра** (`stats_core`, по строке на вариант):
- `likes_only` — учтены только положительные лайки (`like_count > LIKE_THRESHOLD`);
- `likes_plus_comments` — лайки **+** позитивная тональность комментариев (полная
  формула ядра, `is_core`).

Для каждого варианта: `total_users`, `core_users`, доля ядра (`core_share`).

> «Конвертированная тональность комментариев» (поправка тональности комментария
> на тональность поста) в текущей версии реализована как identity (= тональность
> самого комментария); правило конверсии оставлено открытым для уточнения по
> результатам статистики.

## 6. Схема данных

Prisma-модели (`admin-panel/prisma/schema.prisma`):

- `community` — сообщество (регион, город, URL, число подписчиков).
- `post` — публикация; `+ sentiment Sentiment?`, связь `reactions reaction[]`.
- `comment` — комментарий; `sentiment Sentiment?`.
- `like` — бинарный лайк (пост × пользователь).
- `reaction` — **новая**: агрегат реакций поста
  (`reaction_id`, `post_id`, `vk_reaction_id`, `reaction_name`, `sentiment`,
  `count`; `@@unique([post_id, vk_reaction_id])`).
- `user` — пользователь; `+ like_count`, `comment_positive/negative/neutral`,
  `is_core`, пересчитываемый `segment`.
- `stats_sentiment_by_reaction`, `stats_post_sentiment_map`, `stats_core` —
  **новые** материализованные таблицы статистики.

`Sentiment` — enum (`positive` / `negative` / `neutral`);
`Segment` — enum (`Loyal` / `Neutral` / `Disloyal`).

---

## 7. Результат

Сервис автоматически формирует **базу пользователей из сообществ Чувашии** с:
- **меткой лояльности** (сегмент Loyal / Neutral / Disloyal на основе формулы
  ядра);
- **метриками степени и характера вовлечённости** (лайки, тональность
  комментариев, признак ядра);
- **сводной статистикой** по тональности реакций, комментариев и постов
  и по доле ядра — доступной в веб-панели AdminJS.

Итоговая формула лояльности учитывает не только случайный лайк, но и комментарии
пользователей и содержание постов.

## 8. Конфигурация (env)

Ключевые переменные (полный список — `vk-parser/.env.example`):

| Переменная | По умолчанию | Назначение |
|------------|:------------:|------------|
| `VK_TOKEN` | — | Токен VK API (обязателен) |
| `DATABASE_URL` | — | Подключение к Postgres (обязателен) |
| `LLM_BASE_URL` / `LLM_MODEL` / `LLM_API_KEY` | — | Настройка LLM-классификатора |
| `CLASSIFY_CONCURRENCY` | 8 | Число параллельных обращений к классификатору |
| `LIKE_THRESHOLD` | 1 | Порог лайков для ядра |
| `COMMENT_THRESHOLD` | 1 | Порог позитивных комментариев для ядра |
| `CORE_COMBINE` | `and` | Операция объединения условий ядра (`and`/`or`) |
| `SKIP_EXISTING_COMMUNITIES` | true | Пропуск уже собранных сообществ целиком |
| `VK_RATE_LIMIT` | ≈3 | Ограничение частоты VK-вызовов (req/s) |
| `COLLECT_SINCE`, `MAX_POSTS_PER_GROUP`, `MAX_COMMENTS_PER_POST`, `MAX_COMMUNITIES` | — | Границы сбора |
